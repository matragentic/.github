# Contributing to Matragentic

## Welcome

Matragentic is a project built on the belief that agentic systems work better when organised around care, distributed authority, and communal deliberation. That belief extends to how the project itself is built. Contributing here should feel like joining a conversation, not submitting to a process.

This guide covers how to participate across all three repositories — foundation, matrai, and chronicle. Each has a slightly different character, so the ways of contributing differ too.

---

## Before you contribute

**Read the foundation document.** Everything in this project grows from it. Even if you are contributing code to matrai or an observation to the chronicle, the foundation is what gives those contributions context. You do not need to agree with every word, but you should understand what the project is trying to be.

**Understand the pace.** Matragentic values considered responses over fast ones. A pull request that sits for a week while a maintainer thinks about it is not being ignored — it is being taken seriously. If you need a faster response on something time-sensitive, say so and we will try to accommodate.

**You do not need permission to start.** If you see something worth doing — a typo, a gap in the documentation, an idea for the implementation — open an issue or a pull request. You do not need to ask first. If it turns out the contribution needs discussion before merging, we will have that discussion on the PR itself.

## Ways to contribute

### To the foundation

The foundation document is the project's philosophical core. It changes slowly and deliberately, like the purpose layer it describes.

Contributions to the foundation might include:

- **Clarifications** — places where the reasoning is unclear or could be expressed more precisely
- **Deepening** — areas where the thinking is thin and could benefit from more development
- **Challenges** — genuine disagreements with the reasoning, offered in good faith with alternatives proposed
- **References** — anthropological, philosophical, or technical sources that strengthen or complicate the thinking
- **Translations** — the foundation in other languages

Foundation changes are discussed before merging, even small ones. This is not bureaucracy — it is because the foundation shapes everything else, and changes to it deserve communal consideration.

### To matrai

Matrai is the reference implementation — the buildable thing. This is where code, vault structures, agent definitions, and technical documentation live.

Contributions to matrai might include:

- **Bug fixes and improvements** to existing functionality
- **New features** that serve the design philosophy (check the foundation if unsure whether something fits)
- **Documentation** — technical guides, examples, setup instructions
- **Testing** — new tests, edge cases, reliability improvements
- **Vault templates** — starter structures for different kinds of communes
- **Agent definitions** — new agent profiles and specialist knowledge areas
- **Tooling** — scripts, integrations, and utilities that support commune operation

For substantial new features, open an issue first to discuss the approach. This avoids the frustration of building something thoughtful only to learn it conflicts with existing plans or design principles. For smaller changes, a direct pull request is fine.

### To the chronicle

The chronicle is the commune's own account of its experience. Until the commune is running and producing its own chronicle material, contributions here are limited. Once it is, the chronicle will be maintained by the chronicling current within the commune itself.

What is welcome now:

- **Observations** from anyone experimenting with matragentic ideas in their own work
- **Questions** that the chronicle should eventually address
- **Structural suggestions** for how the chronicle should be organised

## How to contribute

### Issues

Issues are how we notice things together. Use them to report problems, propose ideas, surface observations, or ask questions. We have issue templates to help structure different kinds of contributions — use them when they fit, ignore them when they do not.

An issue does not need to be polished. "I noticed something odd when..." or "I wonder whether..." are perfectly good openings. The project values honest observation over formal proposals.

### Pull requests

For changes to any repository:

1. Fork the repository and create a branch with a descriptive name
2. Make your changes, keeping commits focused and messages clear
3. Open a pull request with a description of what you changed and why
4. Be available for conversation — PRs here are collaborative, not transactional

A few norms worth knowing:

**One concern per PR.** A pull request that fixes a bug, adds a feature, and reformats a file is three pull requests. This makes review easier and discussion clearer.

**Explain the why, not just the what.** "Changed X to Y" is less useful than "Changed X to Y because the previous approach caused Z in this situation." The reasoning matters as much as the change, because it helps the community learn.

**Be patient with review.** Maintainers may ask questions, suggest alternatives, or take time to think. This is engagement, not obstruction. If a PR is going to take longer than a week to review, we will tell you.

**Disagreement during review is normal.** If you and a reviewer see something differently, that is a conversation worth having. State your reasoning, listen to theirs, and trust that the goal is the best outcome for the project rather than either person being right.

### Discussions

GitHub Discussions (when enabled) are the right place for open-ended conversation that is not yet an issue or a PR. "What if the tender lifecycle had a sixth state?" or "Has anyone tried running a commune with fewer than five agents?" — these are discussions, not issues.

## What we value in contributions

**Alignment with purpose.** Contributions should serve what the project is trying to be. The foundation document is the reference. If you are unsure whether something fits, ask — that question is itself a valuable contribution.

**Honesty over polish.** A rough idea clearly stated is more useful than a polished proposal that obscures uncertainty. Say what you know, what you think, and what you are unsure about.

**Care for what exists.** Before proposing something new, understand what is already there and why. The project has made deliberate choices; engaging with them — even to disagree — produces better contributions than starting from a blank slate.

**Leanness.** The project values simplicity. A contribution that makes the system simpler while preserving its capabilities is worth more than one that adds capability at the cost of complexity. When in doubt, do less, better.

**Acknowledgement of others.** If your contribution builds on someone else's work, issue, or idea — say so. The project values an accurate record of how understanding develops collectively.

---

## Recognition

Every contribution that is merged will be acknowledged. For the foundation and chronicle, contributors are named in the document's revision history. For matrai, contributors appear in the standard git history and in release notes where appropriate.

Significant contributions — a new section of the foundation, a major feature in matrai, a particularly insightful observation — may be marked in the project's acknowledgements, which mirrors how the commune itself celebrates good work.

---

## A note on the project's stage

Matragentic is young. The foundation is written but the implementation is being built. The first commune has not yet run. This means:

- **There is a lot to do** and many ways to contribute meaningfully
- **Things will change** as building reveals what the design got right and wrong
- **Early contributors shape the culture** of the community that forms around this work

If the ideas in the foundation resonate with you, now is a good time to join. The conversation is still small enough to be genuine.

---

*Questions about contributing? Open an issue in any repository, or reach out to the maintainers. We would rather answer a question than miss a contribution.*
